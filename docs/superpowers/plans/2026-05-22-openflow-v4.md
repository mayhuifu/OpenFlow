# OpenFlow V4 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship V4 — make test results persistent and queryable. Default backend: SQLite (zero-config, file next to the JSON report). Optional backend: PostgreSQL (multi-bench shared). Add a CLI (`openflow report query / trend / ingest / list-sessions / show / migrate`) and a JSON-to-DB migration tool for engineers with archived V1-V3 reports. **No web UI in V4** — that's V5's job.

**Architecture:** New `openflow/report/db/` subpackage with a SQLAlchemy-Core-based schema layer (not ORM — keeps queries SQL-readable, no ORM overhead at the ~1M-rows-per-bench-per-year scale we target). File-based migration scripts in `openflow/report/db/migrations/` (no Alembic dependency). Two backends share most SQL: SQLite quirks live in `sqlite_backend.py`, PostgreSQL quirks in `postgres_backend.py`. Plugin hook writes to local SQLite at sessionfinish; optional shared-DB write is best-effort (failure → warning, not test failure). New `openflow report` CLI subcommands wrap `sqlite_backend` for queries.

**Tech Stack:** Python 3.11 · uv · sqlite3 (stdlib) · sqlalchemy>=2.0 (Core, not ORM) · psycopg2-binary (optional extra) · matplotlib (optional, for `--plot`) · pytest · ruff · mypy · GitHub Actions

Spec: `docs/superpowers/specs/2026-05-22-openflow-v4-design.md`

---

## Prerequisites

V4 should not start until **V3 ships and stabilizes** (a release cycle of bench feedback, ideally one). Reasons:
- Schema design depends on what measurement types V3 brings (each new instrument's measurement methods feed back into per-record column shape).
- The `Path` columns + `model` field shape solidify only after V3 fixture wiring is exercised on a real bench.

If V3 isn't released and bench-validated when V4 starts, **skip Task 1's beta-flag default decision** and ship V4a with `storage.persist: true` from day one — but then plan for a v0.9.1 schema fix-up release.

```python
# openflow/report/db/schema.py
from sqlalchemy import (Column, DateTime, Float, ForeignKey, Integer, JSON,
                        MetaData, String, Table, UniqueConstraint)

metadata = MetaData()

sessions = Table(
    "sessions", metadata,
    Column("session_id", String, primary_key=True),
    Column("started_at", DateTime, nullable=False),
    Column("finished_at", DateTime),
    Column("host", String),
    Column("openflow_version", String),
    Column("config_path", String),
    Column("passed", Integer),
    Column("failed", Integer),
    Column("exit_status", Integer),
)

tests = Table(
    "tests", metadata,
    Column("test_id", Integer, primary_key=True, autoincrement=True),
    Column("session_id", String, ForeignKey("sessions.session_id"), nullable=False),
    Column("test_node_id", String, nullable=False),
    Column("testcase_id", String),
    Column("verdict", String),  # 'passed', 'failed', 'skipped'
    Column("duration_s", Float),
)

cases = Table(
    "cases", metadata,
    Column("case_id", Integer, primary_key=True, autoincrement=True),
    Column("test_id", Integer, ForeignKey("tests.test_id"), nullable=False),
    Column("params", JSON),
    Column("recorded_at", DateTime),
)

measurements = Table(
    "measurements", metadata,
    Column("measurement_id", Integer, primary_key=True, autoincrement=True),
    Column("case_id", Integer, ForeignKey("cases.case_id"), nullable=False),
    Column("name", String, nullable=False),
    Column("value", Float),
    Column("text_value", String),  # for non-numeric (e.g. IDN strings)
    Column("unit", String),
)
```

Plus indexes on `(testcase_id, started_at)` for trend queries.

# Stop conditions

Pause V4 if:

- **Schema design feedback in v0.9.0-alpha is substantive.** That's a signal to do a v0.9.x cycle before flipping the default to `persist: true`.
- **Engineers don't use the CLI.** If the bench team is happy with HTML reports and never opens a terminal for trend analysis, the CLI surface is overdesigned. Drop trend; ship only ingest + query.
- **PostgreSQL setup overhead in the lab is too high.** If no team has an internal Postgres available, V4b is unused infrastructure. Keep the code but flag in docs as "advanced — most teams stick with SQLite."
