# OpenFlow V5 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship V5 — turn OpenFlow from a per-engineer-per-bench tool into team-coordinated lab infrastructure. Three capabilities, each independently shippable: bench reservation, multi-DUT parallel runs, read-only web dashboard. Targeting v1.0.0 — the OpenFlow "version 1" milestone realizing the full vision from the V1 design document.

**Architecture:** Three new subsystems under `openflow/`. `openflow/bench/` owns reservation (file-lock locally; DB-row in shared mode, reading from V4's database). `openflow/parallel/` extends pytest-xdist with a coordinator that locks shared instruments across parallel sessions. `openflow/dashboard/` is a FastAPI + HTMX read-only webapp that reads from V4's database and from a live-heartbeat localhost socket fed by the pytest plugin during test sessions. **No write API** on the dashboard — engineers re-run tests with corrected config rather than clicking edit buttons. Each subsystem ships in its own phase: V5a (reservation) → V5b (parallel) → V5c (dashboard), then v1.0.0.

**Tech Stack:** Python 3.11 · uv · pytest-xdist · filelock · fastapi + uvicorn (dashboard) · jinja2 + htmx (dashboard frontend, no SPA build) · sqlalchemy (queries against V4 DB) · matplotlib (optional, charts in dashboard) · ruff · mypy · GitHub Actions

Spec: `docs/superpowers/specs/2026-05-22-openflow-v5-design.md`

---

## Prerequisites

V5 **strictly depends** on V4 shipping:

- Reservation uses the V4 `sessions` table to record active reservations
- Dashboard reads from V4's tables for historical views
- Live test-progress events are correlated with V4 session IDs

If V4 hasn't shipped or has known schema gaps, **do not start V5**. Loop V4 first.

V5 also depends on **2-3 engineers wanting it**. The reservation feature solves multi-engineer coordination — a one-engineer lab has no use for it. The dashboard solves "what's running where right now" — a one-bench lab has no use for it. Confirm team scale before opening Phase 1.

# Stop conditions

Pause V5 if:

- **Team is still effectively 1-2 engineers.** Reservation has no value at small team scale; multi-DUT may or may not. Single dashboard for the engineer's own data is overengineering — engineer just runs `openflow report query` from the terminal.
- **Engineers reject the read-only constraint.** If the team genuinely needs to edit verdicts via the dashboard (e.g. mark a known-flaky case as "manually approved"), V5's scope balloons. Re-spec before implementing.
- **Multi-DUT design review surfaces a hard deadlock case.** Halt V5b until a robust lock model is in place. Don't ship a system that can hang the whole lab.
- **Dashboard becomes the source of truth for things it shouldn't.** If engineers send dashboard URLs in tickets instead of HTML reports, hold the line: dashboard URLs are ephemeral, HTML is durable.

# Post-1.0 candidates (V6+, out of scope here)

- ML-assisted regression detection
- Multi-bench load balancing / scheduling
- Cloud-hosted shared dashboard for distributed labs
- Test author auto-completion in IDEs (Python LSP-aware fixture introspection)
- Mobile dashboard view
- Slack/Teams notifier integrations
- Test scheduling / queue ("run these 20 tests overnight in this order")

Those become the V6+ conversation, not V5.
